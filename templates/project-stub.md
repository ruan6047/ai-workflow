## 核心鐵律（速查）
1. **實作／審核分離**：同一張卡的執行與查核＝不同任務、不同經手者；**任何人不得自審自己實作的卡**（含 Claude Code）。
2. **分支 + 部署閘門**：**程式碼(A 類)每卡開分支** `ai/<模型或工具>/<卡ID>`（文件/資料維運不必，見 canonical §0.1）；**只有 `main`（已審核合併）能部署，分支不部署**。
3. **獨立性紅線**：紅線卡（安全/金流/統計 ML 正確性/資安/資料正確性）審核**必換模型家族或人審**，且**必跑實測**——同家族審（含 Opus 審 Sonnet）不算數。一般卡同家族異 session 審可接受。
4. **退回不代改**：審核發現缺陷 → 退回 + 缺陷報告 → **原執行者同分支修** → 重審；審核者**不得順手改**。連續 ≥3 次退回 → 升級。
5. **留痕**：commit 加 trailer（**人＝GitHub 帳號如 `ruan6047`，勿寫「使用者」；AI＝`模型@工具`**）：
   ```
   Requested-by:   <GitHub 帳號 | 業務/來源>
   Planned-by:     <GitHub 帳號 | AI/模型>
   Implemented-by: <模型@工具>
   Reviewed-by:    <GitHub 帳號 | 模型@工具>
   ```
6. **通用紅線（canonical §9）**：**防幻覺**（先讀再說、不虛構不存在的 API/表/指令、程式碼為事實文件追之）；**secrets 永不進 git**；**提交 AI 產出者負最終責任**；交付必附「改什麼/為什麼/怎麼驗證(實測)」；不擅升鎖定版本。
7. **狀態自動同步（canonical §3.2）**：交付與部署分欄；GitHub PR merge 後自動記 `📦已合併 + SHA`。需部署的卡只有專案自訂 adapter 回報 `✅已驗證` 才能 `🏁完成`；失敗／回滾不得封存。
8. **模型路由（canonical §10）**：先依風險選 L1–L4，再選供應商；紅線 review 必換家族。生產自動化釘明確 model ID，不用 `latest` alias。

**派工＝人工**（使用者指派）；Coordinator 由使用者指定，未指定才由 Claude Code 擔任。任何工具擔任該卡執行者時都不得兼任查核。
