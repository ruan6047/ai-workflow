## 範圍

來源＝cpbl-analytics 2026-07-15 的一連串實際事故（RECORD-API1 / RECORD-API1-FIX1）。**每一條都是踩到才寫的，不是預想**。

### 缺口 1：worktree 沒有 owner 與收尾時機（ruan6047 指出）
退回 → 開修復卡 → 再開 worktree → 修復卡 merge → **原 worktree 殘留、原分支殘留、原卡停在 `↩退回` 沒人收**，最後由 ruan6047 手動清。原規則 §3.1 只寫「怎麼建、怎麼清」，**沒寫誰負責、什麼時候**。ruan6047 並指出：**若一次查核開出兩張 bug 卡，殘留問題會加倍惡化**。

→ 新增 **§3.1.1 worktree 生命週期**：以**卡族**[card family]（原卡 + 其所有修復卡，卡族 ID＝原卡 ID）為單位，**一卡族一 worktree**；分支仍一卡一條，修復卡在同一 worktree 內切新分支，**worktree 不隨分支增生**。**清理者＝merge 者**，merge 是一組動作、**順序不可調換**（`cd` 出目錄 → 合併 → **移除 worktree** → **刪本地分支** → 刪遠端分支 → 轉態）；PM 以 `git worktree list` 對帳 Ledger 活卡族，殘留即流程缺陷。卡片格式加 `worktree` 欄作為對帳依據（**僅 A 類**，B/C 類填 `—不適用`）。

### 缺口 2：退回流程假設「碼還在分支上」
§5 寫「回原執行者、同分支修」，但碼已 merge 時無此路徑，導致臨場發明流程。**孤兒卡的根源**：修復卡結案了，沒人記得回頭收原卡。

→ 新增 **§5.1 修復卡**（僅事後查核適用）：**一次查核＝一張修復卡**，findings 全部收斂進同一張（嚴禁一 finding 一卡——這正是 ruan6047 擔心的「兩張 bug 卡」情境）；卡 ID＝`<原卡ID>-FIX<n>`，**沿用原卡 worktree**；**原卡不得獨立結案，由修復卡帶動一起結案**；main 帶已知缺陷期間不得部署該功能。
→ 新增 **§5.2 事後查核**：定義為**流程失敗後的補救、非正常路徑**，代價誠實揭露（findings 擋不下合併、原卡須揭露流程缺失、須重建分支與 worktree、是否回退由使用者裁示）。並在 §5 明訂：**退回的預設路徑是同分支修，不新增卡／分支／worktree**。

### 缺口 3：閘門靠自律（三次違規證明無效）
同一執行者 AI 連續三張卡直接 `push origin HEAD:main` 才回報「待查核」，且刪除分支與 worktree。規則白紙黑字寫著 §2/§3，照樣違反。

→ §3 明訂**執行者不得自行 merge／push main**；**branch protection 由「建議」升為「應開啟」**，附教訓：未由平台強制的閘門，實質上不存在。

### 缺口 4：通用工程紅線（三條，皆為實際踩到）
- **§9.3.1 測試先紅後綠（升為通用，原本只綁 BUG 快線）**：新增的回歸測試中有一支**斷言方向寫反、恆真**，靠 `git stash` 掉修復重跑才抓到。假測試比沒測試更糟。
- **§9.3.2 驗證證據須註明環境；新 worktree 先跑基線**：某卡回報「pytest 120 passed」屬實，但那個 shell 恰好帶了 `PYTHONPATH`；乾淨 checkout 下整套 collection 直接失敗——**驗證迴圈失效而所有人以為它是綠的**。
- **§9.3.3 同一判準只寫一份**：「現役球員」判準在同檔寫了兩份，第二份漏一個資料來源 → 已離隊但本季有成績的球員被標成非現役。給查核者的具體提問：「這個判準有沒有第二份拷貝？」
